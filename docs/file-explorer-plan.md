# File Explorer Right Sidebar Plan

## Goal

Add a project file explorer as a right-sidebar window, similar to the existing diff window, using `@pierre/trees`. The explorer should let users browse the current project workspace files, toggle from the keyboard with `mod+shift+e` (`Cmd+Shift+E` on macOS, `Ctrl+Shift+E` elsewhere), and keep behavior predictable for large repositories, reconnects, and missing workspace state.

`@pierre/trees` latest checked during planning: `1.0.0-beta.3`. It exports React bindings from `@pierre/trees/react` and accepts a `FileTree` model created from path-first input.

## Product Behavior

1. The right panel has mutually exclusive windows: diff or file explorer.
2. `mod+d` keeps toggling the diff window.
3. `mod+shift+e` toggles the file explorer.
4. Opening file explorer closes diff by clearing diff-specific route params.
5. Opening diff closes file explorer by clearing explorer-specific route params.
6. On narrow layouts, file explorer uses the same sheet behavior as diff.
7. The explorer root is the active project workspace:
   - Use the project `cwd` / workspace root.
   - Do not switch to `thread.worktreePath`; the tree is project-specific so switching threads in the same project does not refetch or redraw the explorer.
   - Show a non-crashing empty state when no workspace is available.
8. Selecting a file should keep focus inside the explorer experience and show a read-only file preview in the adjacent content area once preview support lands. Until then, selection opens the file in the preferred editor via existing editor/open APIs.
9. Opening a file in the preferred external editor should remain available after preview support lands, but it should move to a deliberate action such as double-click, Enter, or an explicit icon/button so single-click selection can drive preview state.
10. Directory expansion, selection, and search state can remain client-local for the first iteration; URL state should only encode whether the explorer window is open.
11. Git status should be visible in the file tree when the active project is a git repository.
12. When a selected file has working-tree changes, the preview should be able to show those changes inline or in a diff-oriented mode.

## Architecture

### Route & Panel State (‚úÖ done)

`apps/web/src/diffRouteSearch.ts` is the source of truth for right-panel URL state:

- `panel?: "diff" | "files"` ‚Äî which panel is open (mutually exclusive).
- `diff?: "1"` ‚Äî legacy alias accepted for backward compatibility; normalized to `panel: "diff"` on parse.
- `diffTurnId`, `diffFilePath` ‚Äî only populated when diff is the active panel.
- Helpers: `isDiffPanelOpen(search)`, `getOpenRightPanel(search)`, `parseDiffRouteSearch(raw)`, `stripDiffSearchParams(params)`.

The route `apps/web/src/routes/_chat.$environmentId.$threadId.tsx` drives both panels:

- Derives `diffOpen` and `filesOpen` from parsed search.
- Tracks lazy-mount state separately for each panel (`hasOpenedDiff`, `hasOpenedFiles`) so content stays mounted after first open.
- Exposes `openDiff`/`closeDiff` and `openFiles`/`closeFiles` navigation callbacks.
- Inline layout: renders `DiffPanelInlineSidebar` and `FileExplorerPanelInlineSidebar` side by side; only the active one has `open=true`.
- Sheet layout: renders two `RightPanelSheet` instances; only the active one has `open=true`.
- `reserveTitleBarControlInset` is `false` when either panel is open.

### Keybindings (‚úÖ done)

- `packages/contracts/src/keybindings.ts` ‚Äî `"fileExplorer.toggle"` in `STATIC_KEYBINDING_COMMANDS`. The settings UI label "File Explorer: Toggle" is auto-generated by the `commandLabel` formatter.
- `packages/shared/src/keybindings.ts` ‚Äî default binding `{ key: "mod+shift+e", command: "fileExplorer.toggle", when: "!terminalFocus" }`.
- `apps/web/src/keybindings.ts` ‚Äî `isFileExplorerToggleShortcut` helper.
- `apps/web/src/components/ChatView.tsx` ‚Äî handles `fileExplorer.toggle` in the existing window-level keydown handler; calls `onToggleFiles()` which navigates to `panel: "files"` or strips panel params.

**Bug fixed:** `onToggleDiff` and `onToggleFiles` were not closing their panels when the shortcut was pressed a second time. Root cause: `retainSearchParams(["panel", "diff"])` on the route re-adds any params that are merely absent from the new search object. The close path was returning `stripDiffSearchParams(previous)` which omits the keys entirely ‚Äî so the middleware put them back. Fix: explicitly include `panel: undefined, diff: undefined` in the closing navigation so the middleware sees the keys are intentionally cleared.

### FileExplorerPanel (‚úÖ done)

`apps/web/src/components/FileExplorerPanel.tsx` ‚Äî the panel component. Props: `mode?: DiffPanelMode`.

- Reads active project via `useParams` + store selectors, then uses the project `cwd` as the stable explorer root.
- Fetches entries with `useQuery(projectListEntriesQueryOptions({ environmentId, cwd }))`.
- Reuses the existing `useGitStatus({ environmentId, cwd })` subscription for working-tree decorations; no additional polling loop.
- Tracks a client-local selected file path and fetches bounded previews with `projectReadFileQueryOptions({ environmentId, cwd, relativePath })`.
- Converts `ProjectEntry[]` to `@pierre/trees` path strings: directories get a trailing `/`.
- Builds a `FileTree` model once via `useFileTree`; calls `model.resetPaths(paths)` in `useEffect` on entry changes.
- Maps current working-tree changed-file paths to `GitStatusEntry[]` and calls `model.setGitStatus(gitStatusEntries)` when status updates.
- `useFileTree` options: `flattenEmptyDirectories: true`, `initialExpansion: 0`, `search: true`, `dragAndDrop: false`, `renaming: false`, custom `sort` (directories before files, then locale-numeric basename order).
- Themes the shadow-DOM tree via CSS custom properties (`--trees-*-override`) on the wrapper `div`.
- Renders panel states for loading, no workspace, empty workspace, and the live tree plus preview states for no selection, loading, text, binary, too-large, missing, and error.
- Keeps the file tree column compact by default, with a persisted resize handle so the adjacent preview pane gets the remaining panel width unless the user expands the tree.
- Lazy-loaded via `LazyFileExplorerPanel` in the route; renders in both inline-sidebar (`mode="sidebar"`) and sheet (`mode="sheet"`) layouts.

### Tree Library Capabilities

`@pierre/trees@1.0.0-beta.3` supports the planned git decoration work directly:

- `gitStatus?: readonly GitStatusEntry[]`
- `model.setGitStatus(gitStatus)`
- `GitStatus` values: `"added" | "deleted" | "ignored" | "modified" | "renamed" | "untracked"`
- `renderRowDecoration?: (context) => FileTreeRowDecoration | null` for additional badges/icons

`@pierre/trees` does not provide file content preview or file diff rendering. It only provides tree UI, selection state, and row decorations. Preview and diff rendering must be implemented in T3 Code using separate project/file and VCS RPCs plus local React components.

## Data And RPC Plan

### First Iteration

Recommended new RPC:

- Contract: `projects.listEntries`
- Input: `{ cwd: string; limit?: number }`
- Output: `{ entries: ProjectEntry[]; truncated: boolean }`
- Implementation: reuse the internal `WorkspaceEntries` index builder and ignore filtering already used for search.
- Limits: keep `WORKSPACE_INDEX_MAX_ENTRIES` protection, return `truncated`, and avoid blocking the WebSocket server under large repos.

Reasoning: `@pierre/trees` wants path-first full tree input. `filesystem.browse` is shallow; `WorkspaceEntries` already has VCS-aware traversal, so a list API is the cleanest reuse point.

### Later Iteration

If startup cost is noticeable in large repos, switch the explorer to lazy loading:

- Use `filesystem.browse` per expanded directory.
- Feed incremental add/reset operations into the `@pierre/trees` model.
- Cache expanded directory results with React Query and invalidate on checkpoint rollback/revert hooks.

### Preview And Diff Data

Implemented RPCs / data sources:

- `projects.readFile`: reads a project-relative text file for preview. Reads are bounded by size, path-validated, realpath-contained under the project root, and return explicit non-crashing states for binary, too-large, and missing files.

Planned RPCs / data sources:

- `vcs.getFileStatus` or an extension to `vcs.status`: expose per-path status kinds (`added`, `deleted`, `modified`, `renamed`, `untracked`, `ignored`) instead of only insertion/deletion counts.
- `vcs.getFileDiff`: fetch the current working-tree diff for one project-relative file. This should cover staged and unstaged changes, and distinguish deleted/renamed/untracked files.

Initial shortcut:

- Use the existing `useGitStatus({ environmentId, cwd: projectCwd })` result for changed-file paths.
- Map all changed files to tree status `"modified"` until the server exposes richer per-file status kinds.
- Use insertion/deletion counts only for optional row decorations or preview summaries.

### Milestone 5 ‚Äî File selection and edge states (‚úÖ done)

1. ‚úÖ **File activation ‚Üí open in editor.** `@pierre/trees` does not expose a distinct "activate" (double-click / Enter) event; the public surface is `onSelectionChange?: (selectedPaths: readonly string[]) => void`. `FileExplorerPanel` now wires selection changes to `openInPreferredEditor` when the selected path is a file. It resolves paths with `resolvePathLinkTarget(relativePath, projectCwd)` so editor opens are project-root relative and stable across thread switches.
2. ‚úÖ **Directory activation** ‚Äî handled natively by the tree; no extra wiring needed.
3. ‚úÖ **drag/drop and rename** already disabled (`dragAndDrop: false`, `renaming: false`) in `FileExplorerPanel`.
4. ‚úÖ **Truncated banner** ‚Äî `truncated` is already surfaced in the header row. No additional work needed.
5. ‚úÖ **Project-specific tree root.** The explorer intentionally uses project `cwd`, not `thread.worktreePath`, so the query key and tree model are project-specific instead of thread-specific.
6. ‚úÖ **Query invalidation on workspace change** ‚Äî different project `cwd` already produces a different React Query key so the query refetches automatically. Checkpoint rollback/revert currently invalidates `projectQueryKeys.all` through the environment connection service, which refreshes the explorer list along with other project queries.

## Reliability Notes

- Do not scan arbitrary absolute paths from the client. Only list the resolved active project workspace.
- Treat explorer data as best-effort UI state ‚Äî a scan failure must not break chat rendering or WebSocket session state.
- Keep all paths relative in the web UI; only the server and editor-open call deal with absolute paths.
- File preview and diff RPCs must validate that requested paths stay inside the active project workspace.
- Preview reads must be size-bounded and should avoid loading binary files into React state.
- Git status decorations are advisory UI state. Missing or stale git status must not block tree rendering.
